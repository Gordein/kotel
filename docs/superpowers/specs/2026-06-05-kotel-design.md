# Котёл — сплиттер расходов на квартиру (design)

- **Дата:** 2026-06-05
- **Статус:** согласовано в брейншторме, ждёт ревью спеки
- **Рабочее имя:** Котёл (общий котёл; легко переименовать)

## 1. Контекст и цель

Трое снимают квартиру в Польше: **Сэм**, **Люда**, **Микита**. Платят друг за
друга — аренда, коммуналка, продукты, хозтовары — и в конце месяца сводят итоги.
Сейчас это ведётся в Google-таблице вручную (видно по присланным скринам:
лист «Арэнда» и лист «Расходы» с колонками на каждого и заметками).

**Цель:** заменить таблицу на PWA, где каждый со своего телефона вносит траты и
переводы, а приложение само считает «кто кому и сколько должен», ведёт общую
ленту событий и позволяет оставлять комментарии — **без перегруза интерфейса**.

Это самостоятельный проект, не связанный с geointel. Живёт в отдельном репозитории
`C:\Users\latas\Downloads\kotel`.

## 2. Люди и вход

- Один общий «дом» = ваша квартира. Внутри **три профиля** (Сэм, Люда, Микита). Без мульти-тенанта.
- Вход: **только по PIN**, без выбора имени. PIN уникален и сам определяет, в чей кабинет
  ты входишь (по умолчанию Сэм=111, Люда=222, Микита=333). PWA запоминает устройство
  (cookie), дальше открываешь и сразу ты. Без email и паролей.
- Сменить **свой** PIN — в профиле. Чужой PIN никто не трогает (функции сброса чужого нет).
- **Текущая личность видна** в шапке, чтобы случайно не внести трату от чужого имени.

## 3. Валюта

Только **злотый (zł / PLN)**. Одна валюта на всё. Без конвертаций и курсов
(мультивалюта сознательно выкинута — источник споров и ошибок). Суммы — 2 знака
после запятой (гроши).

## 4. Категории

Фиксированный набор чипов, выбор тапом (не свободный текст — иначе фрагментация):

`Квартира` · `Продукты` · `Хозтовары` · `Связь` · `Другое`

Аренда и коммуналка — одна категория **«Квартира»**. По умолчанию «Продукты». Набор
задаётся в коде; редактирование из UI — на будущее.

## 5. Модель данных

Пять сущностей. Ключевая идея: у траты есть **кто заплатил** (один или несколько) и
**на кого делится** (любое подмножество троих). Это закрывает и обычную пятёрку за
продукты, и вашу аренду в две руки, и случай «Сэм этого не ел».

| Сущность | Поля |
|---|---|
| **Person** | id, name, color, pin_hash, created_at. Ровно 3 строки. |
| **Expense** | id, title, amount (= Σ payers, каноничный тотал), category, spent_on (дата), note, created_by_id, created_at, updated_at, **version**, **request_id** (idempotency), deleted_at (soft-delete). + строки `payers` и `shares`. |
| **ExpensePayer** | expense_id, person_id, amount. Кто реально платил. Обычно 1 строка; у аренды — 2. |
| **ExpenseShare** | expense_id, person_id, amount. На кого делим (разрешённые суммы, не веса — округление зафиксировано при создании). |
| **Settlement** | id, from_person_id, to_person_id, amount, method (cash/transfer), settled_on, note, created_by_id, created_at, version, request_id, deleted_at. Это «Микита закрыл долг Люде». |
| **Comment** | id, target_type (expense/settlement), target_id, author_id, text, created_at, deleted_at. Одна таблица на оба типа целей. |
| **Template** | id, title, category, default_payers (json), default_shares (json), note (напр. чек-лист переводов Михалу), active. Повторяющиеся траты (аренда, коммуналка) — добавляются в один тап. |

`ExpensePayer`/`ExpenseShare` — это строки одной сущности Expense, не новые
концепты. Лента активности — **не отдельная таблица**, собирается из Expense +
Settlement + Comment по времени (не плодим данные).

## 6. Основные операции

- **Добавить трату.** Простой путь (90% случаев): сумма, название, категория. Дефолты:
  плачу я, делим поровну на всех троих, дата сегодня → три тапа. «Разделить иначе» —
  разворачивает выбор плательщиков и долей/исключений.
- **Закрыть долг.** Кому, сколько, нал/перевод. Баланс пересчитался, в ленте
  «Сэм отдал Люде 700 zł наличкой».
- **Шаблоны.** Кнопки «Аренда»/«Коммуналка» сверху формы — создают готовую трату за
  выбранный месяц одним тапом.
- **Править / удалить.** Любой правит любую запись (вы свои люди), но всё именное и
  пишется в ленту («Люда изменила трату»). Удаление — **мягкое** (deleted_at), запись
  остаётся в истории как «удалено», чтобы не осиротить комментарии и не сломать ленту.
- **Комментарий.** На любой трате/переводе. В ленте свёрнуто (`💬 2`), тап — развернулись.
- **Баланс.** Всегда актуальный «кто кому должен сейчас» + подсказка минимальных
  переводов, чтобы всё обнулить.
- **Период.** Бегущий баланс (долги переносятся, пока не закрыты). Фильтр по месяцу —
  только для просмотра «итогов за июнь», на сам долг не влияет.

## 7. Баланс и расчёт «кто кому должен»

Балансы **не хранятся**, а считаются на чтении из неизменяемых записей (см. §9).
Для каждого человека `p`:

```
paid_out(p) = Σ выплат p в тратах (ExpensePayer) + Σ переводов, отправленных p
benefit(p)  = Σ долей p в тратах (ExpenseShare) + Σ переводов, полученных p
net(p)      = paid_out(p) − benefit(p)
```

- `net(p) > 0` → группа должна `p`; `net(p) < 0` → `p` должен группе.
- **Инвариант:** `Σ net(p) == 0` всегда (замкнутая система). Это самопроверка —
  если сумма не ноль, где-то баг; показываем предупреждение, а не молча врём.
- **Подсказка переводов:** жадный матчинг крупнейший должник → крупнейший кредитор,
  пока все ≈0. Для троих даёт минимум переводов и совпадает с попарной картиной.

## 8. Проверка модели на вашей аренде

Шаблон «Аренда» за июнь в один тап создаёт трату:
- **amount 5200**, payers **{Люда: 2600, Микита: 2600}**, shares **{Сэм: 1900, Люда: 1900, Микита: 1400}**.

Расчёт `net`: Сэм `0 − 1900 = −1900`; Люда `2600 − 1900 = +700`; Микита `2600 − 1400 = +1200`
(сумма = 0 ✓). Подсказка: **Сэм → Люде 700, Сэм → Миките 1200** (= его доля 1900).

К шаблону приложен **чек-лист переводов Михалу** (заметка, на баланс не влияет):
- ☐ Люда → Михал: Najem 1900, Opłaty 700
- ☐ Микита → Михал: Najem 1900, Opłaty 700

→ те самые «4 перевода». Сэм отдаёт нал → два Settlement (Сэм→Люда 700, Сэм→Микита 1200),
балансы обнулились.

## 9. Анализ ошибок и защита от них

Главное требование: **ничьи данные не теряются и не «скидываются»**, даже когда двое
заполняют формы одновременно. Подход — по лучшим практикам.

### 9.1. Параллельная работа (главный страх)

| Ситуация | Что происходит | Защита |
|---|---|---|
| Двое одновременно добавляют **разные** траты | Оба INSERT независимы, баланс выводится из строк → обе появляются | Неизменяемые записи + баланс на чтении. Потерять нечего by design. |
| Ты заполняешь форму, в это время сосед что-то добавил, лента/баланс обновились | Фоновое обновление трогает **только** области ленты/баланса | Форма — изолированный клиентский черновик (Alpine). Фоновый refresh её не стирает. Сабмит уходит как есть. |
| Двое правят **одну и ту же** трату | Второй сейв не проходит молча | **Оптимистичная блокировка:** `UPDATE … WHERE id=? AND version=?`. 0 строк → «Люда только что изменила эту трату, обнови и сохрани заново». |
| Двойной тап / ретрай сети на мобиле | Дубль не создаётся | **Idempotency-ключ** (`request_id`, UUID с клиента) уникален; повтор игнорируется. |
| Сосед удаляет трату, пока ты её комментируешь | Комментарий не осиротеет | **Мягкое удаление**: строка остаётся, коммент сохранён, показываем «трата удалена». |
| Перевёл по устаревшему балансу (сосед параллельно добавил трату) | Перевод — просто факт оплаты, не зависит от показанного тотала | Баланс пересчитается с обеими записями; покажем «баланс изменился, пока ты переводил». Деньги не теряются, остаток до-настраивается. |

**Технически:** SQLite в режиме **WAL** + `busy_timeout` — записи сериализуются (один
писатель за раз, читатели не блокируются), для троих это пуленепробиваемо.

### 9.2. Ошибки ввода

| Ошибка | Защита |
|---|---|
| Сумма плательщиков ≠ тотал | В простой форме тотал = одна сумма одного плательщика — рассинхрону неоткуда взяться. В режиме нескольких плательщиков тотал = Σ payers (вычисляется). |
| Доли не сходятся с тотал (100/3 = 33.33×3 = 99.99) | Равный делёж с остатком по методу наибольшего остатка; последний грош добивается. Показываем разрешённые суммы — всегда сходится. |
| Исключили из траты всех | Блок: минимум один участник. |
| Сумма ≤ 0 | Блок: сумма > 0. |
| Перевод самому себе | Блок: from ≠ to. |
| Перевод больше долга (переплата) | Мягкое подтверждение: «Микита должен Люде только 50, ты вводишь 100 — Люда станет должна 50. Точно?» |
| Не та дата/месяц | Дефолт — сегодня; дата редактируется; месяц виден в форме. |
| **Семантический дубль** (двое внесли аренду за июнь; одну покупку записали дважды) | Шаблон проверяет «трата из этого шаблона за июнь уже есть» и предупреждает. Для обычных — мягкая подсказка «похоже, такое уже добавляли» при совпадении сумма+название за короткий интервал. |
| Не тот плательщик (по умолчанию «я», а платила Люда) | Поле редактируемо; в ленте видно и кто создал, и кто платил. |
| Чужая личность на чужом телефоне | Текущая личность видна в шапке, смена в один тап. |

### 9.3. Бухгалтерская корректность

| Риск | Защита |
|---|---|
| Накопление ошибок округления | Остаток фиксируется при создании, хранятся разрешённые суммы; балансы всегда сводятся к 0. |
| Правка/удаление траты, которая уже «закрыта» переводами | Балансы пересчитываются ретроактивно (правильно), но это может «оживить» закрытый долг. Показываем «баланс изменился из-за правки прошлой траты», событие — в ленту. Молча не рассинхронизируем. |
| Путаница «месяц vs всего» | Текст в UI: баланс — это «сейчас всего», вкладка месяца — «что было в июне». Фильтр не меняет долг. |
| Время/таймзона ломает порядок ленты | Серверное время (UTC хранится, Europe/Warsaw показывается). Клиентские часы не влияют. |
| Сумма балансов не ноль (скрытый баг) | Инвариант `Σ net == 0` проверяется и показывается как ошибка, а не прячется. |

## 10. Экраны (mobile-first, раскладка как у Claude, без перегруза)

Логика Claude: спокойный фон, узкая центральная колонка, контент карточками,
действие — внизу под большим пальцем. Принцип против перегруза — **прогрессивное
раскрытие**: сложное (доли, граф, комментарии, чек-лист) свёрнуто, разворачивается по тапу.

Четыре экрана, переключение — нижний таб-бар: **Баланс · Лента · ＋ · Профиль**.

```
┌─────────────────┐   ┌─────────────────┐
│  Баланс  (Сэм)⚙ │   │  Лента          │
│                 │   │                 │
│ Ты должен:      │   │ Сэм · Продукты  │
│  Люде     700 zł│   │  35 zł · на всех│
│  Миките  1200 zł│   │  💬 1           │
│ Тебе должны: —  │   │ ─────────────── │
│                 │   │ Микита → Люде   │
│ [△ показать граф]│   │  700 zł · нал   │
│                 │   │ ─────────────── │
│ Чтобы закрыть:  │   │ Люда · Аренда   │
│  Сэм→Люда 700   │   │  июнь · 5200 zł │
│  Сэм→Мик. 1200  │   │                 │
│                 │   │                 │
│ [ Закрыть долг ]│   │                 │
└──┬────┬────┬───┬┘   └──┬────┬────┬───┬┘
  Баланс Лента ＋ Профиль   (нижний таб-бар)
```

- **Баланс** — **эго-центричный**: только «ты должен / тебе должны». Граф (по тапу) рисует
  **только твои рёбра** — чужие долги между двумя другими не показываются. Кнопка «Закрыть
  долг» активна, только если ты кому-то должен.
- **Лента** — общие операции хронологически, всё именное, комментарии свёрнуты.
- **＋** — добавить трату: сумма/название/категория + **кто заплатил** и **делим на** —
  всё видно сразу (дефолт: плачу я, делим на всех), с живой подсказкой «по X на каждого». Шаблоны сверху.
- **Профиль** — смена своего PIN, выход.

## 11. Стек и PWA

- **Flask + HTMX + Alpine.js + SQLite (WAL)**. Server-rendered, не SPA — комфорт-стек,
  один язык, легко поддерживать. Для троих SQLite более чем достаточно; ни Postgres,
  ни Redis, ни Celery.
- HTMX — точечные обновления ленты/баланса (короткий polling ~10 c и/или после действий);
  Alpine — мелкие интеракции (развернуть доли/комментарии/граф).
- Граф — лёгкий SVG/Chart, без тяжёлых библиотек.
- **PWA:** manifest + service worker, устанавливается на телефон, кэширует оболочку и
  последнее состояние для быстрого открытия. Запись требует сети (офлайн-запись с
  merge-синхронизацией — сознательно вне v1, чтобы не плодить конфликты).
- Деплой: один контейнер (как привычно) или локально в домашней сети.

## 12. Вне scope (YAGNI)

## 14. Раунд 2 — изменения UX (после догфудинга)

Поверх исходного дизайна внесены правки по фидбеку, источник истины теперь — `CLAUDE.md`:

- **Один экран** вместо вкладок: сверху запись траты (главное действие), ниже баланс + лента.
  Профиль, отдельная страница ленты и граф — убраны.
- **Плательщик = текущий пользователь** (каждый ведёт себя). Селектор «кто платил» убран.
- **Комментарии-треды убраны**; вместо них необязательное `note` на трате, разворачивается в ленте.
- **Удаление своих записей** (мягкое, только автор).
- Лента **по месяцам** с итогом за месяц и временем (Europe/Warsaw, хранение в UTC).
- **Автополлинг** баланса/ленты (15с) + точка на новых записях.
- **Долги — попарные по нетто внутри траты**: для каждой траты неттим (заплатил − доля),
  недоплатившие должны переплатившим пропорционально переплате; суммируем по парам. Повседневный
  делёж симметричен (каждый должен плательщику свою долю), аренда чистая (Сэм → Люде 700,
  Миките 1200, без долга между ними). Заменили min-transfer (перетасовывал/прятал долги).
- **Лента масштабируется**: один месяц за раз (стрелки ‹ ›) + фильтр по человеку. Текстовый
  поиск добавляли, но убрали по фидбэку (путал) — месяц+человек достаточно.
- **Неравные доли** добавляли (режим «разные доли»), затем убрали по фидбэку: позволяли вносить
  за другого, а кейс «скидываемся» решается тем, что каждый вносит свою часть сам (делёж поровну).
- **Бренд**: отображаемое имя приложения — «Orlowicza»; Микита переименован в «Мiкiта».
- **Редактирование траты**: экран «править» на своей трате (только автор), с оптимистичной
  блокировкой версии — при параллельной правке второй получает 409 «открой заново».
- **Шаблон аренды убран** (сущность Template, кнопка-prompt, сервис): аренду вносят как
  обычную трату категории «Квартира», каждый — свою часть. Меньше сущностей, меньше путаницы.
- Категории: `Квартира · Продукты · Хозтовары · Другое` (убрана «Связь»).
- Спокойная палитра (low-contrast), без эмодзи. Старт пустой.
- Хостинг: бесплатно — PythonAnywhere / Oracle Always-Free / дома+Cloudflare Tunnel (см. `DEPLOY.md`).
  Fly.io отклонён (нет постоянного фри). Cloudflare Workers не подходит (нет диска под SQLite).
